# Quality Report Contract

Contract Version: 1.0.0  
Effective Date: 2026-06-30  
Status: Approved

## 1. Purpose and Scope
This document defines the mandatory quality report generated by every agent for every material output. The report provides readiness, risk visibility, and supervisor decision support.


## 2. Applicability
- Every agent stage that produces or modifies an artifact must publish a quality report.
- A quality report must reference the evaluated artifact versions.
- No stage may emit a final Success completion event without an associated quality report reference.

## 3. Quality Report Schema
Required fields:
- qualityReportId
- workflowId
- correlationId
- executionId
- agentId
- reportVersion
- generatedAt
- evaluatedArtifactRefs
- completeness
- consistency
- confidenceScore
- traceability
- openQuestions
- knownRisks
- recommendations
- validationSummary
- approvalRecommendation
- overallStatus
- aiUsage

Mandatory markdown sections (all must exist):
- Validation Summary
- Coverage Summary
- OpenLog Summary
- AI Usage Summary
- Confidence Score
- Readiness
- Blocking Issues

Compactness rules:
- Compact the content, never compact the schema.
- Keep field values concise (1-3 lines where appropriate).
- AI Usage is metadata-only and must not include narrative explanations.

Optional fields:
- exceptionRequests
- followUpActions

Field semantics:
- completeness: Coverage percentage and missing elements summary.
- consistency: Internal and cross-artifact consistency assessment.
- confidenceScore: Integer range 0-100.
- traceability: Mapping quality from outputs to source requirements and artifacts.
- openQuestions: Unresolved items affecting certainty or release risk.
- knownRisks: Explicit risk log entries with impact and likelihood.
- recommendations: Actionable recommendations for next stage.
- validationSummary: Condensed validation category results.
- approvalRecommendation: AutoApproved | MinorReview | RequiresApproval | Blocked.
- overallStatus: Ready | Minor Review | Needs Approval | Blocked.
- aiUsage: Lightweight stage-level AI usage record with optional workflow aggregate/totals.

AI usage subfields (populate with "N/A" or "Estimated" when unavailable):
- workflowId
- correlationId
- agentName
- stageName
- modelName
- modelProvider
- sessionId
- startTime
- endTime
- duration
- inputTokens
- outputTokens
- totalTokens
- estimatedCost
- retryCount
- status
- blockingReason

Supervisor aggregate fields:
- agent
- model
- duration
- input
- output
- totalTokens
- estimatedCost
- status

Workflow totals fields:
- totalAgents
- successful
- failed
- blocked
- totalDuration
- totalInputTokens
- totalOutputTokens
- totalTokens
- estimatedTotalCost

## 4. Threshold Contract
- 90-100: Ready
- 80-89: Minor Review
- 60-79: Needs Approval
- below 60: Blocked

Rules:
1. confidenceScore determines default overallStatus.
2. Critical validation failures force overallStatus to Blocked regardless of numeric score.
3. Any unresolved high-impact open question forces at least Needs Approval.

## 5. Risk Entry Schema
Required fields:
- riskId
- description
- impact: Low | Medium | High | Critical
- likelihood: Low | Medium | High
- mitigation
- owner

Optional fields:
- dueDate
- dependencyRef

## 6. Traceability Requirements
Quality report must include trace links:
- source artifact references and versions
- requirement references
- acceptance criteria references where applicable
- validation result reference

Traceability minimum:
- 100 percent of mandatory outputs linked to at least one source requirement or approved exception.

## 7. Validation Summary Structure
Required categories in summary:
- inputValidation
- outputValidation
- artifactValidation
- businessValidation
- schemaValidation
- traceabilityValidation
- qualityValidation
- guardrailsValidation

Each category includes:
- result: Pass | Warning | Fail
- notes
- blocking: true | false

## 8. Approval Recommendation Rules
- AutoApproved:
  - overallStatus is Ready
  - no critical or high unresolved risks
  - no blocking validation categories
- MinorReview:
  - overallStatus is Minor Review
  - only low or medium risks unresolved
- RequiresApproval:
  - overallStatus is Needs Approval
  - policy, risk, or ambiguity requires human decision
- Blocked:
  - overallStatus is Blocked
  - mandatory correction required before continuation

## 9. Publication and Lifecycle
- Owner agent publishes quality report as immutable artifact version.
- Lifecycle: Draft -> Validated -> Published -> Archived.
- Semantic versioning applies to report structure and interpretation.

## 10. Example Structure
```yaml
qualityReport:
  qualityReportId: qr-backend-0142
  workflowId: wf-2026-06-30-001
  executionId: exec-0142
  agentId: backend-developer
  reportVersion: 1.0.0
  generatedAt: 2026-06-30T12:42:55Z
  evaluatedArtifactRefs:
    - artifactId: art-backend-spec-0011
      version: 1.1.0
  completeness:
    score: 96
    missing: []
  consistency:
    score: 91
    findings: []
  confidenceScore: 92
  traceability:
    coverage: 100
    gaps: []
  openQuestions: []
  knownRisks:
    - riskId: R-19
      description: "Unusually high load burst may degrade latency"
      impact: Medium
      likelihood: Medium
      mitigation: "Enable adaptive queue throttling"
      owner: devops-release
  recommendations:
    - "Proceed to QA with load-test focus"
  validationSummary:
    inputValidation: { result: Pass, notes: "", blocking: false }
    outputValidation: { result: Pass, notes: "", blocking: false }
    artifactValidation: { result: Pass, notes: "", blocking: false }
    businessValidation: { result: Pass, notes: "", blocking: false }
    schemaValidation: { result: Pass, notes: "", blocking: false }
    traceabilityValidation: { result: Pass, notes: "", blocking: false }
    qualityValidation: { result: Pass, notes: "", blocking: false }
    guardrailsValidation: { result: Pass, notes: "", blocking: false }
  approvalRecommendation: AutoApproved
  overallStatus: Ready
```

## 11. Supervisor Consumption Rules
Supervisor must:
1. Verify quality report existence for each stage completion.
2. Enforce threshold-based routing:
   - Ready -> continue automatically.
   - Minor Review -> continue with warning.
   - Needs Approval -> route to approval flow.
   - Blocked -> halt and request corrective action.
3. Persist quality decision in workflow memory and emit corresponding lifecycle and quality events.
